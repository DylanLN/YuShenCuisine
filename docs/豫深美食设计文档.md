# 1 项目背景

深圳聚集大量河南籍人口。

目前寻找河南菜馆主要依赖：

* 高德地图
* 美团
* 大众点评
* 微信群推荐

存在以下问题：

* 店铺分散
* 搜索效率低
* 无专题地图
* 无河南菜聚合平台

因此建设：

**深圳河南美食地图**

统一收录深圳市全部河南餐馆。

---

# 2 项目目标

建设一个可在 Github Pages 免费运行的网站。

用户能够：

* 查看深圳河南餐馆分布
* 搜索河南美食
* 查看店铺详情
* 查看店铺位置
* 一键导航

无需注册登录。

---

# 3 产品定位

本项目是：

**河南人在深圳找家乡味道的平台。**

不是：

* 外卖平台

# 4 数据采集策略

## 采集方式

**一次性全量采集 + 手动更新。**

1. 使用高德 Web API `v5/place/text` 进行一次全量搜索
2. 搜索结果保存为静态 JSON 文件
3. 后续数据更新由作者手动执行脚本并提交

**不设置自动采集任务。** 不去定时调用高德 API，以避免触发限流或产生费用。

## 采集行政区

福田、罗湖、南山、盐田、宝安、龙岗、龙华、坪山、光明、大鹏

共 10 个区（去掉深汕合作区，名称不统一且实际价值低）。

## 采集关键词

河南菜、河南饭店、河南餐馆、河南面馆、河南烩面、郑州烩面、羊肉烩面、胡辣汤、
开封灌汤包、洛阳水席、信阳菜、周口菜、驻马店菜、安阳菜

## API 调用方案

每次请求参数：`keywords` + `region`（行政区），一次一页（page_size=25）。

对每个关键词 × 每个行政区发起请求，如有分页则逐页获取。

预估总请求量：14 个关键词 × 10 个区 × 约 2 页 ≈ 280 次请求，远低于免费额度 5,000 QPD。

## 采集流程

```
行政区 + 关键词
     ↓
高德 Web API v5/place/text
     ↓
解析 JSON 响应
     ↓
提取字段（名称、地址、电话、坐标、评分等）
     ↓
数据清洗（去除无效/重复记录）
     ↓
去重（优先 poi_id，次选 店名+地址）
     ↓
生成 crawler/output/restaurants.json
     ↓
运行整理脚本 scripts/sync-data.sh
（将数据复制到 web/data/ 并生成 districts.json / categories.json）
```

## 去重规则

优先使用 `poi_id`；如果不存在，使用 `店名 + 地址` 作为唯一键。

## API Key 管理

* 高德 Web API Key 配置在 `crawler/config.py` 中
* 仅在本地开发环境使用，不提交到仓库

# 5 地图方案

**使用静态地图图片，不接入高德 JS API。**

理由：

* 零 API 调用，零费用，零 Key 管理
* Github Pages 可直接加载，无跨域问题
* 满足查看店铺分布的核心需求

## 实现方式

1. 使用一张深圳地图截图（OpenStreetMap / 高德静态图）作为底图
2. 标记点通过 CSS 绝对定位叠加在底图上
3. 坐标 → 像素的转换通过预计算 bbox + 线性插值实现
4. 点击标记弹出简讯（店名、评分），点击可跳转详情页

## 标记聚合策略

| 店铺数量 | 展示方式 |
|----------|----------|
| < 50 家 | 全部标记直接展示 |
| 50 ~ 200 家 | 按行政区聚合，点击区名展开 |
| > 200 家 | 按行政区聚合 + 只显示有店铺的区 |

## 详情页位置展示

显示地址文本 + "导航到店" 按钮（跳转高德地图 APP / 高德 Web 版），不再展示地图图片。

### 字段说明

| 字段 | 类型 | 必需 | 说明 | 来源 |
|------|------|------|------|------|
| id | string | 是 | 高德POI ID | API 返回 |
| name | string | 是 | 店铺名称 | API 返回 |
| district | string | 是 | 所属行政区（短名称） | API 返回 |
| address | string | 是 | 详细地址 | API 返回 |
| phone | string | 否 | 联系电话，无则 `""` | API 返回 |
| longitude | number | 是 | 经度 | API 返回 |
| latitude | number | 是 | 纬度 | API 返回 |
| rating | number | 否 | 评分，无则 `null` | API 返回 |
| cost | number | 否 | 人均消费，无则 `null` | API 返回 |
| tags | string[] | 是 | 分类标签（至少一个） | 采集时推断 |
| cover | string | 否 | 封面图片 URL，无则 `""` | 预留字段 |
| source_keyword | string | 是 | 采集时命中的关键词 | 采集时记录 |
| update_time | string | 是 | 数据更新时间 | 采集时生成 |

# 9 标签推断规则（TAG_RULES）

采集时，将店名与关键词规则表匹配，自动生成 `tags` 字段。这条规则写死在 `crawler/config.py` 中，不依赖 AI 猜测。

```python
TAG_RULES = {
    "烩面": "烩面",
    "胡辣汤": "胡辣汤",
    "灌汤包": "灌汤包",
    "开封灌汤包": "灌汤包",   # 精确匹配，优先于 "开封" → "河南菜"
    "羊肉汤": "羊肉汤",
    "河南菜": "河南菜",
    "豫菜": "河南菜",
    "信阳": "河南菜",
    "周口": "河南菜",
    "驻马店": "河南菜",
    "安阳": "河南菜",
    "洛阳": "河南菜",
    "开封": "河南菜",
    "扣碗": "河南菜",
}
```

匹配逻辑：

```python
def infer_tags(name: str, source_keyword: str) -> list:
    tags = set()
    for keyword, tag in TAG_RULES.items():
        if keyword in name:
            tags.add(tag)
    if not tags:
        tags.add("河南菜")
    return list(tags)
```

# 13 数据流设计

## 整体数据流

```
data/*.json
    │
    ▼
Pinia Store ──→ 组件读取
(restaurant.ts)
    │
    ├──→ 列表页（全量遍历 + 分页）
    ├──→ 地图页（按可视区域/行政区过滤）
    ├──→ 详情页（按 id 查找）
    └──→ 搜索/筛选（客户端过滤，防抖 300ms）
```

## Pinia Store 设计

```typescript
interface RestaurantStore {
  // 状态
  all: Restaurant[]          // 全量数据
  loading: boolean           // 加载状态
  error: string | null       // 错误信息

  // 筛选条件
  filterDistrict: string     // 区域筛选（'' 为全部）
  filterCategory: string     // 分类筛选（'' 为全部）
  searchKeyword: string      // 搜索关键词

  // 计算属性
  filtered: Restaurant[]     // 按条件过滤后的结果
  districts: string[]        // 可用行政区列表
  categories: string[]       // 可用分类列表

  // 操作
  fetchAll(): Promise<void>                    // 从 data/restaurants.json 加载
  getById(id: string): Restaurant | undefined  // 按 id 查找单个店铺
}
```

## 地图数据交互

1. Pinia Store 加载 `data/restaurants.json`
2. `MapPage.vue` 读取 `store.filtered` 作为数据源
3. 将数据传入 `StaticMap.vue` 组件
4. `StaticMap.vue` 根据店铺坐标计算像素位置，渲染标记点

# 18 错误处理与边界状态

## 数据加载

| 状态 | 处理方式 |
|------|----------|
| 加载中 | 骨架屏 / 加载动画 |
| 加载失败 | 显示错误提示 + 重试按钮 |
| 数据为空 | 显示 "暂无店铺数据" 引导文案 |

## 搜索与筛选

| 场景 | 处理方式 |
|------|----------|
| 无搜索结果 | "没有找到相关店铺，试试其他关键词" |
| 区域无店铺 | 置灰不可选区域 / 提示"该区域暂无收录" |

## 地图

| 场景 | 处理方式 |
|------|----------|
| 底图加载失败 | 显示 "地图加载失败"，展示纯列表 |
| 无坐标数据 | 标记点不显示，店铺在列表页可查 |
| 坐标超出范围 | 忽略该标记，日志记录 |

## 坐标转换函数

```typescript
interface BBox {
  minLng: number
  maxLng: number
  minLat: number
  maxLat: number
}

function lngLatToPixel(
  lng: number, lat: number,
  bbox: BBox, imgWidth: number, imgHeight: number
): { x: number; y: number } {
  const x = ((lng - bbox.minLng) / (bbox.maxLng - bbox.minLng)) * imgWidth
  const y = ((bbox.maxLat - lat) / (bbox.maxLat - bbox.minLat)) * imgHeight
  return { x, y }
}
```

## 坐标系说明

高德 API 返回的是 **GCJ-02** 坐标系（国测局加密坐标）。

| 底图来源 | 坐标系 | 是否需要转换 |
|----------|--------|-------------|
| 高德地图截图 | GCJ-02 | 不需要，直接匹配 |
| OpenStreetMap 截图 | WGS-84 | 需要 GCJ-02 → WGS-84 转换 |

**推荐使用高德地图截图作为底图**，这样 API 返回的坐标无需任何转换即可直接映射到图片上，完全避免偏移问题。

如果未来因版权等原因需要使用 OSM 底图，在 `crawler/crawl.py` 中添加 GCJ-02 → WGS-84 转换函数即可，前端代码无需改动。
